## 1. Data Fetching — TanStack Query

The web client (`apps/web`) uses **TanStack Query v5** (`@tanstack/react-query`) for all server-state management. TanStack Query owns the remote cache; no Zustand/Redux store is needed for server data. Local UI state (wizard step, draft, toast queue) lives in React state or `useReducer`.

### 1.1 Typed API client generated from the OpenAPI/Zod spec

The backend exposes a fully generated OpenAPI document at `GET /api/v1/openapi.json` (derived from the shared `@stabil/types` Zod schemas — see [architecture/04-api-contracts.md](../architecture/04-api-contracts.md) §1.2). A build-time script in `apps/web` generates a typed fetch client from this spec:

```ts
// apps/web/src/lib/api/client.ts
import createFetchClient from "openapi-fetch";
import type { paths } from "./openapi"; // generated by openapi-typescript

export const apiClient = createFetchClient<paths>({
  baseUrl: process.env.NEXT_PUBLIC_API_BASE_URL ?? "http://localhost:3001/api/v1",
});
```

`openapi-typescript` + `openapi-fetch` together give end-to-end type safety: every path, method, request body, and response shape is inferred from the spec — if the backend Zod schema changes and the spec is regenerated, TypeScript surfaces breakages at build time.

> **Generating the client:** `pnpm turbo generate:api-types` in `apps/web` calls
> `openapi-typescript http://localhost:3001/api/v1/openapi.json -o src/lib/api/openapi.d.ts`.
> Run this after any backend contract change and commit the updated `.d.ts`.

### 1.2 Auth token handling and refresh interceptor

The access token (15 min lifetime, see [architecture/04-api-contracts.md](../architecture/04-api-contracts.md) §1.3) is kept in an in-memory variable inside the token store module — never in `localStorage`. The `HttpOnly` cookie carries the refresh token on web; the store holds only the access token.

```ts
// apps/web/src/lib/api/token-store.ts
let accessToken: string | null = null;
let accessTokenExp: number = 0; // unix seconds

export function setTokens(token: string, expiresIn: number) {
  accessToken = token;
  accessTokenExp = Math.floor(Date.now() / 1000) + expiresIn;
}

export function clearTokens() {
  accessToken = null;
  accessTokenExp = 0;
}

export function getAccessToken(): string | null {
  return accessToken;
}

export function isExpired(): boolean {
  // treat as expired 30 s before actual exp to avoid races
  return accessTokenExp - 30 < Math.floor(Date.now() / 1000);
}
```

The refresh interceptor wraps `apiClient` with middleware that silently refreshes before any request whose stored token is within the 30-second expiry window:

```ts
// apps/web/src/lib/api/auth-middleware.ts
import { apiClient } from "./client";
import { getAccessToken, isExpired, setTokens, clearTokens } from "./token-store";

let refreshPromise: Promise<void> | null = null;

async function silentRefresh(): Promise<void> {
  // POST /auth/refresh — refresh token is in the HttpOnly cookie; no body needed on web.
  const { data, error } = await apiClient.POST("/auth/refresh", { body: {} });
  if (error || !data) {
    clearTokens();
    // Redirect to /login — handled by router middleware
    throw new Error("session-expired");
  }
  setTokens(data.accessToken, data.expiresIn);
}

export const authMiddleware: Parameters<typeof apiClient.use>[0] = {
  async onRequest({ request }) {
    if (isExpired()) {
      // Deduplicate concurrent refresh calls with a shared promise
      refreshPromise ??= silentRefresh().finally(() => { refreshPromise = null; });
      await refreshPromise;
    }
    const token = getAccessToken();
    if (token) {
      request.headers.set("Authorization", `Bearer ${token}`);
    }
    return request;
  },
};

apiClient.use(authMiddleware);
```

Any `401 unauthenticated` or `401 token-reuse-detected` response from the server (see [architecture/04-api-contracts.md](../architecture/04-api-contracts.md) §1.5) is caught by a TanStack Query global `onError` callback that redirects to `/login` and calls `clearTokens()`.

### 1.3 Query key conventions

Query keys follow a hierarchical tuple convention so that invalidation is surgical and predictable:

```ts
// apps/web/src/lib/query-keys.ts
export const qk = {
  // Profiles
  profile:        (profileId: string)  => ["profiles", profileId]             as const,
  myProfiles:     ()                   => ["profiles", "mine"]                as const,

  // Submissions
  submission:     (profileId: string, mode: string) =>
                    ["profiles", profileId, "submissions", mode]               as const,

  // Score runs
  scoreRun:       (runId: string)      => ["scoring", "runs", runId]          as const,
  scoreHistory:   (profileId: string)  => ["scoring", "runs", { profileId }]  as const,

  // Report
  report:         (profileId: string)  => ["profiles", profileId, "report"]   as const,

  // Documents
  documents:      (profileId: string)  => ["documents", { profileId }]        as const,

  // Verification
  verification:   (profileId: string)  => ["verification", { profileId }]     as const,

  // Notifications
  notifications:  ()                   => ["notifications"]                   as const,
} as const;
```

The top-level resource name is always the first element. This lets `queryClient.invalidateQueries({ queryKey: ["profiles", profileId] })` invalidate everything scoped to a profile in one call.

### 1.4 Caching and staleness policy

| Query | `staleTime` | `gcTime` | Notes |
|-------|-------------|----------|-------|
| `report` | 60 s | 5 min | Re-scored data should feel live but re-fetches are not free. |
| `submission` | 30 s | 5 min | Drafts are saved explicitly; stale draft reads are fine for short windows. |
| `scoreHistory` | 2 min | 10 min | Historical runs change only after a re-score. |
| `documents` | 15 s | 3 min | Upload confirmation makes the status volatile. |
| `myProfiles` | 60 s | 5 min | Low-write, shown in nav/sidebar. |
| `notifications` | 10 s | 2 min | Expected to feel near-real-time. |

```ts
// apps/web/src/lib/query-client.ts
import { QueryClient } from "@tanstack/react-query";

export const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 30_000,   // 30 s global default — overridden per hook as above
      gcTime:   300_000,   // 5 min
      retry: (failureCount, error) => {
        // Do not retry auth errors or 4xx (except 429)
        const status = (error as any)?.status;
        if (status && status !== 429 && status >= 400 && status < 500) return false;
        return failureCount < 2;
      },
    },
  },
});
```

```ts
import { apiClient } from "@/lib/api/client";
import { qk } from "@/lib/query-keys";
```

```ts
export function useSubmission(profileId: string, mode: Mode) {
```

### 1.6 Mutations and optimistic updates

Mutations use `useMutation`. For the submission save (which happens in the background after every wizard step advance), we apply an **optimistic update** so the UI never shows stale data:

```ts
// apps/web/src/hooks/use-save-submission.ts
import { useMutation, useQueryClient } from "@tanstack/react-query";
import { apiClient } from "@/lib/api/client";
import { qk } from "@/lib/query-keys";
import type { Mode, SubmissionAnswers } from "@stabil/types";

export function useSaveSubmission(profileId: string, mode: Mode) {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: async (answers: SubmissionAnswers) => {
      const { data, error } = await apiClient.PUT(
        "/profiles/{profileId}/submissions/{mode}",
        {
          params: { path: { profileId, mode } },
          body: { answers },
        },
      );
      if (error) throw error;
      return data;
    },

    onMutate: async (answers) => {
      // Cancel any in-flight refetches so they don't overwrite our optimistic data
      await queryClient.cancelQueries({ queryKey: qk.submission(profileId, mode) });
      const previous = queryClient.getQueryData(qk.submission(profileId, mode));

      // Optimistically merge the new answers
      queryClient.setQueryData(qk.submission(profileId, mode), (old: any) =>
        old ? { ...old, answers: { ...old.answers, ...answers } } : null,
      );

      return { previous };
    },

    onError: (_err, _answers, context) => {
      // Roll back on failure
      if (context?.previous !== undefined) {
        queryClient.setQueryData(qk.submission(profileId, mode), context.previous);
      }
    },

    onSettled: () => {
      // Always reconcile with the server after settle
      queryClient.invalidateQueries({ queryKey: qk.submission(profileId, mode) });
    },
  });
}
```

### 1.7 Triggering a re-score and invalidating the report

After a candidate saves new answers and triggers a score run, the report cache must be invalidated so it refreshes automatically. The sequence is:

1. `PUT /profiles/:profileId/submissions/:mode` — save answers
2. `POST /scoring/runs` — trigger score run (requires `Idempotency-Key`)
3. Invalidate `report` and `scoreHistory` keys so TanStack Query refetches fresh data

```ts
// apps/web/src/hooks/use-run-score.ts
import { useMutation, useQueryClient } from "@tanstack/react-query";
import { apiClient } from "@/lib/api/client";
import { qk } from "@/lib/query-keys";
import { v7 as uuidv7 } from "uuid";

export function useRunScore() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: async (profileId: string) => {
      const idempotencyKey = uuidv7();
      const { data, error } = await apiClient.POST("/scoring/runs", {
        headers: { "Idempotency-Key": idempotencyKey },
        body: { profileId },
      });
      if (error) throw error;
      return data;
    },

    onSuccess: (data) => {
      const { profileId } = data;
      // Invalidate the cached report (it now reflects an older run)
      queryClient.invalidateQueries({ queryKey: qk.report(profileId) });
      // Invalidate score history (new run appears at the top)
      queryClient.invalidateQueries({ queryKey: qk.scoreHistory(profileId) });
      // Seed the new run into the cache immediately (avoids a redundant fetch)
      queryClient.setQueryData(qk.scoreRun(data.id), data);
    },
  });
}
```

> **Idempotency-Key:** the client generates a fresh UUID v7 per mutation invocation. If the network fails and the user retries, `useMutation`'s `retry` logic will call `mutationFn` again with a **new** key — correct behaviour, since a network-failure retry is semantically a new attempt. Do NOT share a key across retries unless you are explicitly replaying.

## 2. Forms — react-hook-form + Zod

All forms use **react-hook-form v7** with a **Zod resolver** (`@hookform/resolvers/zod`). The Zod schemas live in `packages/types` and are imported by both the web client and the NestJS API, guaranteeing identical validation at both layers (SCOPE §10 "Validation: Zod (shared schemas)").

### 2.1 Shared Zod schemas from `packages/types`

```ts

  cloudExposure:        z.number().int().min(1).max(5).optional(),
```

## 3. Multi-Step Wizard Pattern

Both the **Fresher** and **Working Professional** intake flows are multi-step wizards. The wizard is the centrepiece of Phase 1 (see [pages/mode-selection-and-forms.md](./pages/mode-selection-and-forms.md) for the exact step inventory and per-mode field listings).

### 3.1 Step definition

Each step is a typed object that the wizard engine drives. Putting step metadata in data (rather than in JSX trees) makes it easy to add, reorder, or conditionally skip steps:

```ts
// apps/web/src/features/wizard/types.ts
import type { z } from "zod";
import type { ComponentType } from "react";

export interface WizardStep<TData = unknown> {
  /** Stable slug used as a URL segment and localStorage key */
  id: string;
  /** Display label for the progress indicator */
  label: string;
  /** Zod schema validating ONLY the fields collected on this step */
  schema: z.ZodTypeAny;
  /** React component rendered for this step */
  component: ComponentType<{
    defaultValues?: Partial<TData>;
    onNext: (data: TData) => void;
    onBack: () => void;
  }>;
  /** If true, the step is shown but all fields are optional — user may skip */
  optional?: boolean;
}
```

### 3.2 Draft autosave and resume-where-left-off

The wizard continuously persists the in-progress answers to **localStorage** under a namespaced key (`stabil:wizard:draft:<profileId>:<mode>`). On mount, it hydrates from localStorage before fetching from the server. The server draft (from `GET /profiles/:profileId/submissions/current`) is the fallback and the source of truth after any re-open; localStorage is a client-side cache that prevents data loss on accidental tab closes.

```ts
// apps/web/src/features/wizard/use-wizard-draft.ts
import { useState, useEffect, useCallback } from "react";
import type { SubmissionAnswers, Mode } from "@stabil/types";

const DRAFT_VERSION = 1; // bump to invalidate stale drafts after schema changes

function draftKey(profileId: string, mode: Mode) {
  return `stabil:wizard:draft:v${DRAFT_VERSION}:${profileId}:${mode}`;
}

export function useWizardDraft(profileId: string, mode: Mode, serverDraft: SubmissionAnswers | null) {
  const key = draftKey(profileId, mode);

  const [draft, setDraft] = useState<SubmissionAnswers>(() => {
    if (typeof window === "undefined") return serverDraft ?? {};
    try {
      const stored = localStorage.getItem(key);
      return stored ? JSON.parse(stored) : (serverDraft ?? {});
    } catch {
      return serverDraft ?? {};
    }
  });

  // Keep localStorage in sync whenever draft changes
  useEffect(() => {
    try { localStorage.setItem(key, JSON.stringify(draft)); } catch { /* quota exceeded — ignore */ }
  }, [key, draft]);

  const mergeDraft = useCallback((patch: Partial<SubmissionAnswers>) => {
    setDraft((prev) => ({ ...prev, ...patch }));
  }, []);

  const clearDraft = useCallback(() => {
    localStorage.removeItem(key);
    setDraft(serverDraft ?? {});
  }, [key, serverDraft]);

  return { draft, mergeDraft, clearDraft };
}
```

**Resume-where-left-off:** the wizard stores the current step index alongside the draft (`stabil:wizard:step:<profileId>:<mode>`). On re-open, it reads this index, validates that the preceding steps' saved data passes schema, and either resumes at the stored step or falls back to step 0.

### 3.4 Step validation gating

A user cannot advance to step N+1 until step N's Zod schema passes. The wizard controller enforces this:

```ts
// apps/web/src/features/wizard/use-wizard.ts
import { useState, useCallback } from "react";
import type { WizardStep } from "./types";
import type { SubmissionAnswers } from "@stabil/types";

export function useWizard(steps: WizardStep[], initialStep = 0) {
  const [stepIndex, setStepIndex] = useState(initialStep);

  const current = steps[stepIndex];

  const next = useCallback(async (stepData: unknown) => {
    // Validate the current step's schema before advancing
    const result = await current.schema.safeParseAsync(stepData);
    if (!result.success) return { success: false, error: result.error };
    if (stepIndex < steps.length - 1) setStepIndex((i) => i + 1);
    return { success: true };
  }, [current, stepIndex, steps.length]);

  const back = useCallback(() => {
    setStepIndex((i) => Math.max(0, i - 1));
  }, []);

  const jumpTo = useCallback((index: number) => {
    // Only allow jumping backwards (to already-completed steps)
    if (index < stepIndex) setStepIndex(index);
  }, [stepIndex]);

  return { stepIndex, step: current, next, back, jumpTo, isFirst: stepIndex === 0, isLast: stepIndex === steps.length - 1 };
}
```

### 3.5 The Review step

The final wizard step is a **read-only summary** of all collected answers before the user submits. It lists every answer grouped by section, highlights any empty optional fields, and gives the user a chance to jump back and correct anything. The "Submit & Score" button on the Review step:

1. Calls `PUT /profiles/:profileId/submissions/:mode` with the full merged draft.
2. On success, calls `POST /scoring/runs` (with a fresh `Idempotency-Key`).
3. Invalidates the `report` and `scoreHistory` query keys (see §1.7).
4. Clears the localStorage draft.
5. Navigates to the candidate report dashboard.

**Phase 2 extension — "Review extracted data":** when resume parsing is live (Phase 2), the server suggests pre-filled values from the LLM parse job (via OpenRouter). A dedicated variant of the Review step shows the AI-extracted answers alongside a confidence indicator and lets the candidate confirm, edit, or reject each suggestion before they are merged into the draft. This variant renders the same `WizardStep` `component` prop interface with an additional `parsedSuggestions?: ParsedSuggestions` prop; the wizard controller passes it only when a parse result is available (see [architecture/01-overview.md](../architecture/01-overview.md) §4.6).

## 4. File Uploads via Presigned MinIO URLs

File bytes **never transit the NestJS API** — the client PUTs them directly to MinIO using a short-lived presigned URL (SCOPE §10 "Storage"; [architecture/04-api-contracts.md](../architecture/04-api-contracts.md) §9; [backend/modules/documents-storage.md](../backend/modules/documents-storage.md)). The three-step flow is:

**Step 1 — Request a presigned URL**

```ts
const { data: uploadMeta, error } = await apiClient.POST("/documents/upload-url", {
  body: {
    profileId,
    kind,       // "resume" | "certificate" | "gov-id" | "transcript" | "other"
    fileName:    file.name,
    contentType: file.type,
    sizeBytes:   file.size,
  },
});
if (error) throw error;
// uploadMeta: { documentId, uploadUrl, objectKey, expiresAt, requiredHeaders }
```

**Step 2 — PUT bytes directly to MinIO**

```ts
const putResponse = await fetch(uploadMeta.uploadUrl, {
  method: "PUT",
  headers: {
    ...uploadMeta.requiredHeaders,   // e.g. { "Content-Type": "application/pdf" }
  },
  body: file,
});
if (!putResponse.ok) {
  throw new Error(`MinIO PUT failed: ${putResponse.status}`);
}
```

**Step 3 — Confirm to the API**

```ts
const { data: doc, error: confirmError } = await apiClient.POST(
  "/documents/{id}/confirm",
  { params: { path: { id: uploadMeta.documentId } }, body: {} },
);
if (confirmError) throw confirmError;
// doc.status → "uploaded" (then async "scanning" / "ready")
```

The entire flow wrapped as a React hook with progress:

```ts
// apps/web/src/hooks/use-upload-document.ts
import { useMutation, useQueryClient } from "@tanstack/react-query";
import { useState } from "react";
import { apiClient } from "@/lib/api/client";
import { qk } from "@/lib/query-keys";
import type { DocumentKind } from "@stabil/types";

export function useUploadDocument(profileId: string) {
  const queryClient = useQueryClient();
  const [progress, setProgress] = useState(0); // 0..100

  const mutation = useMutation({
    mutationFn: async ({ file, kind }: { file: File; kind: DocumentKind }) => {
      setProgress(10);

      // 1. Request presigned URL
      const { data: meta, error: urlError } = await apiClient.POST("/documents/upload-url", {
        body: {
          profileId,
          kind,
          fileName:    file.name,
          contentType: file.type,
          sizeBytes:   file.size,
        },
      });
      if (urlError) throw urlError;
      setProgress(30);

      // 2. PUT to MinIO — use XHR for progress events
      await new Promise<void>((resolve, reject) => {
        const xhr = new XMLHttpRequest();
        xhr.upload.addEventListener("progress", (e) => {
          if (e.lengthComputable) {
            setProgress(30 + Math.round((e.loaded / e.total) * 55));
          }
        });
        xhr.addEventListener("load", () => {
          if (xhr.status >= 200 && xhr.status < 300) resolve();
          else reject(new Error(`MinIO PUT ${xhr.status}`));
        });
        xhr.addEventListener("error", reject);
        xhr.open("PUT", meta.uploadUrl);
        Object.entries(meta.requiredHeaders).forEach(([k, v]) => xhr.setRequestHeader(k, v));
        xhr.send(file);
      });
      setProgress(90);

      // 3. Confirm
      const { data: doc, error: confirmError } = await apiClient.POST(
        "/documents/{id}/confirm",
        { params: { path: { id: meta.documentId } }, body: {} },
      );
      if (confirmError) throw confirmError;
      setProgress(100);
      return doc;
    },

    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: qk.documents(profileId) });
      queryClient.invalidateQueries({ queryKey: qk.verification(profileId) });
    },

    onSettled: () => { setTimeout(() => setProgress(0), 1500); },
  });

  return { ...mutation, progress };
}
```

**Constraints enforced client-side before requesting the URL:**

| Constraint | Value | Where defined |
|---|---|---|
| Max file size | 10 MB | `NEXT_PUBLIC_MAX_UPLOAD_BYTES` env var; also enforced server-side (`413`) |
| Allowed MIME types | `application/pdf`, `image/png`, `image/jpeg` | Validated before calling `POST /documents/upload-url`; also enforced server-side (`415`) |
| Presigned URL expiry | ~5 min | API returns `expiresAt`; client checks before PUT (re-requests if stale) |

See [pages/documents-and-verification.md](./pages/documents-and-verification.md) for the full upload UI, document-list component, and verification status display. See [backend/modules/documents-storage.md](../backend/modules/documents-storage.md) for MinIO bucket policies, virus-scan integration, and object lifecycle.

## 5. Error Handling

The backend always returns errors as **RFC 9457 `application/problem+json`** (see [architecture/04-api-contracts.md](../architecture/04-api-contracts.md) §1.5). The frontend maps these to two distinct user-facing surfaces depending on the error type:

- **Field errors** (status `422`, `errors[]` populated) → mounted below each input via react-hook-form's `setError`.
- **Non-field errors** (all other statuses) → toast notification via shadcn/ui `useToast`, with a detail message derived from the problem+json `detail` or `title`.

### 5.1 Problem+json type

```ts
// packages/types/src/errors.ts  (re-exported from @stabil/types)
export interface FieldError {
  path:    string;  // e.g. "answers.totalExperienceYears"
  message: string;
  code:    string;  // Zod issue code
}

export interface ProblemDetails {
  type:       string;
  title:      string;
  status:     number;
  detail?:    string;
  instance?:  string;
  requestId?: string;
  errors?:    FieldError[];
}
```

### 5.2 Central error handler

```ts
// apps/web/src/lib/api/handle-api-error.ts
import type { UseFormSetError, FieldPath } from "react-hook-form";
import type { ProblemDetails } from "@stabil/types";
import { toast } from "@/components/ui/use-toast";

/**
 * Maps a problem+json error to field errors (if 422) and/or a toast.
 *
 * Pass `setError` only when called from a form mutation — it is optional
 * so the same function can handle non-form mutations (uploads, run-score, etc.).
 */
export function handleApiError<TFieldValues extends Record<string, unknown>>(
  problem: ProblemDetails,
  setError?: UseFormSetError<TFieldValues>,
): void {
  if (problem.status === 422 && problem.errors?.length && setError) {
    // Map server-side Zod field paths → react-hook-form field names
    for (const fe of problem.errors) {
      // API paths use dot-notation: "answers.totalExperienceYears"
      // RHF paths also use dot-notation — direct mapping works
      setError(fe.path as FieldPath<TFieldValues>, {
        type:    "server",
        message: fe.message,
      });
    }
    // Also show a brief toast so the user knows something is wrong
    toast({ variant: "destructive", title: "Please fix the highlighted fields." });
    return;
  }

  // Non-field errors: map well-known slugs to friendly titles
  const friendlyTitles: Record<string, string> = {
    "unauthenticated":       "Your session has expired. Please sign in again.",
    "forbidden":             "You don't have permission to do that.",
    "consent-required":      "The candidate hasn't shared their report with you yet.",
    "share-expired":         "This share link has expired.",
    "rate-limited":          "You're doing that too quickly. Please wait a moment.",
    "payload-too-large":     "That file is too large (max 10 MB).",
    "unsupported-media-type":"Only PDF, PNG, and JPEG files are accepted.",
    "internal":              "Something went wrong on our end. Please try again shortly.",
    "upstream-unavailable":  "A dependency is temporarily unavailable. Please retry in a moment.",
  };

  const slug = problem.type.split("/").pop() ?? "";
  const title = friendlyTitles[slug] ?? problem.title;

  toast({
    variant: "destructive",
    title,
    description: problem.detail,
  });
}
```

### 5.3 Global TanStack Query error handler

For queries (reads), errors are surfaced via an `onError` in the global `QueryClient` configuration, which calls `handleApiError` without a `setError`:

```ts
// apps/web/src/lib/query-client.ts  (additions)
export const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      // ...staleTime / gcTime / retry as in §1.4...
    },
  },
  queryCache: new QueryCache({
    onError: (error) => {
      const problem = error as ProblemDetails;
      if (problem?.status === 401) {
        // Handled by auth-middleware (redirect to /login); skip toast
        return;
      }
      handleApiError(problem);
    },
  }),
  mutationCache: new MutationCache({
    onError: (error) => {
      const problem = error as ProblemDetails;
      if (problem?.status === 401) return;
      // Mutations that use setError call handleApiError themselves;
      // this global handler covers mutations that don't (e.g. run-score, upload-confirm).
      if (problem?.status !== 422) {
        handleApiError(problem);
      }
    },
  }),
});
```

### 5.4 Per-status guidance summary

| Status | Problem type slug | User-facing treatment |
|---|---|---|
| `422` | `validation-failed` | Field errors under each input via `setError`; brief "fix highlighted fields" toast |
| `401` | `unauthenticated` | Silent redirect to `/login` (auth middleware) |
| `401` | `token-reuse-detected` | Redirect to `/login` + "Security alert" toast |
| `403` | `forbidden` | Destructive toast with context-specific message |
| `403` | `consent-required` | Toast: "The candidate hasn't shared their report yet" |
| `404` | `not-found` | In-page empty state (not a toast); query returns `null` |
| `409` | `conflict` / `idempotency-key-conflict` | Toast: retry the action |
| `410` | `share-expired` | Toast + "Request a new share" CTA |
| `413` | `payload-too-large` | Toast: "File is too large (max 10 MB)" |
| `415` | `unsupported-media-type` | Toast: "Only PDF, PNG, JPEG accepted" |
| `429` | `rate-limited` | Toast with `Retry-After` countdown |
| `500` | `internal` | Toast: generic "something went wrong" |
| `503` | `upstream-unavailable` | Toast: "Service temporarily unavailable — retry shortly" |

## 7. Conventions Cheat-Sheet

| Concern | Convention |
|---|---|
| Query keys | Hierarchical tuples from `qk.*` in `src/lib/query-keys.ts` |
| Default staleTime | 30 s; override per hook for high-volatility data |
| Mutation idempotency | Fresh UUID v7 per `mutationFn` invocation for `POST /scoring/runs` |
| Token storage | In-memory access token; HttpOnly cookie for refresh token |
| Zod schemas | Shared from `@stabil/types`; per-step schemas are `pick`s of the full schema |
| Form mode | `mode: "onBlur"` for wizard steps; `mode: "onChange"` for single-field live-validation flows |
| Draft key | `stabil:wizard:draft:v<DRAFT_VERSION>:<profileId>:<mode>` |
| Draft version | Increment `DRAFT_VERSION` in `use-wizard-draft.ts` on breaking schema changes |
| Error surface | `422` → field errors via `setError`; all others → toast |
| File upload | 3-step (URL → PUT → confirm); XHR for progress; max 10 MB; PDF/PNG/JPEG only |
| Non-transiting files | Bytes never go through the NestJS API — presigned MinIO PUT only |
